Opensense's role: Mid-transport processor. Strips inbound DKIM (body will be modified), adds signature/banner, hands back to M365/Google which re-signs DKIM outbound.

01b — The SMTP Handshake
What two mail servers actually say to each other

Every email delivery is a conversation between two servers. This is that conversation — verbatim. HoverTap any row for what it means.

📤 Sending Server
mail.sender.com
⟷
📥 Receiving Server
mx.recipient.com
Phase 1 — TCP Connection & Greeting
←
220 mx.recipient.com ESMTP ready Server announces itself and readiness
220 — Service Ready
The receiving server opens the connection and identifies itself. The 220 code means "I'm here and ready." If you see anything other than 220, the connection failed before it started.
Phase 2 — EHLO Handshake (Capability Exchange)
EHLO mail.sender.com Sender identifies itself
→
EHLO — Extended Hello
The sending server introduces itself by hostname. EHLO (not old HELO) requests the receiving server's list of supported extensions. The hostname here should match the sending server's PTR record — mismatches can trigger spam flags.
←
250-mx.recipient.com Hello
250-SIZE 52428800
250-STARTTLS
250-AUTH LOGIN PLAIN
250 ENHANCEDSTATUSCODES Receiving server lists what it supports
250 — Capability List
The receiving server responds with what it supports. Key items: SIZE (max message size), STARTTLS (TLS encryption available), AUTH (authentication methods). If STARTTLS is listed, the next step is usually to upgrade to TLS before sending credentials or mail.
Phase 3 — TLS Encryption (STARTTLS)
STARTTLS Request to upgrade to encrypted connection
→
STARTTLS
The sending server requests that the connection be upgraded to TLS encryption. Without this, the entire SMTP conversation including message content travels in plain text. Opensense requires TLS on its connectors — this is that negotiation.
←
220 Ready to start TLS TLS handshake begins — connection is now encrypted
220 — TLS Accepted
From this point on the connection is encrypted. A new EHLO is sent after TLS is established. Everything after this point — credentials, envelope addresses, message content — is encrypted in transit.
Phase 4 — Envelope Exchange (Who / Where)
MAIL FROM:<sender@sender.com> Envelope sender — SPF checks this
→
MAIL FROM — The Envelope Sender
This is the Return-Path / envelope sender. It's NOT necessarily the From address the recipient sees. SPF validates this domain against the sending IP. Bounces are returned here. This can differ from the visible From header — that difference is at the heart of DMARC alignment.
←
250 OK Sender address accepted
250 OK — Sender Accepted
The receiving server accepts the envelope sender. At this point it may run an SPF check against the MAIL FROM domain. Some servers do this here; others wait until after DATA.
RCPT TO:<recipient@recipient.com> Envelope recipient — actual delivery address
→
RCPT TO — The Envelope Recipient
The actual delivery address. Can differ from the To: header (e.g., BCC recipients, mailing lists, aliases). The receiving server checks whether this address exists. A 550 here means the user doesn't exist on that server.
←
250 OK Recipient address accepted — mailbox exists
250 OK — Recipient Accepted
The receiving server confirms the mailbox exists and will accept mail. If it returned 550 here, the NDR you'd see says "user unknown" or "mailbox not found."
Phase 5 — Message Transmission (DATA)
DATA Ready to send the actual message
→
DATA Command
The sending server announces it's about to transmit the full message — all headers and body. The receiving server must respond with 354 before the content is sent.
←
354 Start input, end with "." Go ahead, end message with a period on its own line
354 — Send the Message
The receiving server is ready. The sender now transmits all headers (From, To, Subject, DKIM-Signature, Received, etc.) followed by the message body. The message ends with a single period on its own line.
Message transmitted (headers + body):
From: Jane Smith <sender@sender.com>
To: recipient@recipient.com
Subject: Q2 Planning Meeting
DKIM-Signature: v=1; a=rsa-sha256; d=sender.com; s=selector1; ...
Date: Mon, 20 May 2024 12:15:28 -0400
Message-ID: <CABxxxxxxx@mail.sender.com>

[message body here]

.   ← single period signals end of message
←
250 OK: Message queued as abc123 Message accepted — now in delivery queue
250 OK — Message Accepted
The receiving server accepted the full message. It now runs spam filters, content checks, and authentication verification (SPF, DKIM, DMARC). The queue ID lets admins trace the message in server logs.
Phase 6 — Connection Close
QUIT Done — closing connection
→
QUIT
The sending server cleanly closes the connection. A proper QUIT is the sign of a well-behaved mail server. Servers that drop connections without QUIT can trigger spam signals on some receiving systems.
←
221 Bye Connection closed
221 — Closing Connection
The receiving server closes the TCP connection. The SMTP session is complete. The message is now in the receiving server's queue for local delivery to the mailbox.
MAIL FROM
The envelope sender. SPF checks this. Can differ from From: header. This is the Return-Path.
From: header
What the recipient sees. DMARC aligns against this. Set inside DATA, not the envelope.
RCPT TO
Actual delivery address. Can differ from To: header. BCC recipients appear only here.
DATA section
All headers + body live here. DKIM signs this content. Authentication checks run after DATA is received.
02 — Authentication Checks
SPF · DKIM · DMARC · ARC
SPF
Is this server allowed to send for this domain?
Checks: envelope MAIL FROM
Against: sending IP
Lives in: DNS TXT record
Max lookups: 10
DKIM
Was this message signed and unmodified?
Checks: cryptographic signature
Against: public key in DNS
Breaks if: body is modified
Selector: s=value in header
DMARC
Do SPF/DKIM align with the visible From?
Requires: SPF OR DKIM pass
PLUS: domain alignment
Policy: none/quarantine/reject
Reports: rua= email address
ARC
Preserves auth results across forwarding
Used when: message is forwarded
Preserves: original SPF/DKIM results
Added by: each hop in chain
Helps: legitimate forwarders
Authentication Results
passCheck passed
failHard failure
softfailSPF ~all
temperrorDNS timeout
permerrorConfig broken
noneNo record found

Delivered-To

Delivered-To
Final delivery address. Gmail confirms it delivered here. Useful when checking forwarding or aliasing issues.
INFO
Authentication-Results
dkim=pass header.i=@sender.com header.s=selector1; spf=pass smtp.mailfrom=sender@sender.com; dmarc=pass (p=QUARANTINE) header.from=sender.com
Authentication-Results
THE most important header. Shows SPF, DKIM, and DMARC results. Read this first on every deliverability issue. All three pass here — message is fully authenticated.
CRITICAL
DKIM-Signature
v=1; a=rsa-sha256; d=sender.com; s=selector1; h=from:to:subject:date; bh=Abc123==; b=XyzSig==
DKIM-Signature
d= is the signing domain (must align with From for DMARC).
s= is the selector — find the public key at selector1._domainkey.sender.com
bh= is the body hash — if the body was modified, this won't match.
IMPORTANT
Received
from mail.sender.com (mail.sender.com [203.0.113.45]) by mx.google.com with SMTPS; Mon, 20 May 2024 09:15:31 -0700
Received (hop)
Read ALL Received headers bottom-to-top to trace the message path. "from" = who sent it. "by" = who received it. The IP in brackets is what SPF validates against.
CRITICAL — Read Bottom to Top
Received
from [192.168.1.105] (unknown [203.0.113.45]) by mail.sender.com (Postfix); Mon, 20 May 2024 12:15:29 -0400
Received (origin)
The BOTTOM-MOST Received header is where the message started. [192.168.1.105] is the sender's local machine. (203.0.113.45) is the public IP — this is what SPF validated.
ORIGIN HOP
Return-Path
<sender@sender.com>
Return-Path (Envelope Sender)
This is the MAIL FROM used during SMTP transmission. It's the envelope sender — what SPF actually checks. It may differ from the visible From header. Bounces go here.
SPF CHECKS THIS
From
Jane Smith <sender@sender.com>

What the recipient SEES. This is what DMARC alignment checks against — the d= domain in DKIM and the MAIL FROM domain must align with this domain.
DMARC ALIGNS TO THIS

<CABxxxxxxxxxxxxxxx@mail.gmail.com>
Message-ID
Unique identifier assigned by the originating server. The domain after @ often reveals the actual mail server that generated the message. Useful for cross-referencing logs.

To
recipient@gmail.com

The visible recipient. This can differ from the RCPT TO envelope address. Important in BCC, forwarding, and alias scenarios. Not what delivery is based on.

Email Headers — SPF Fail / DMARC Fail
Authentication-Results
spf=fail (domain of sender@company.com does NOT designate 198.51.100.23 as permitted sender) smtp.mailfrom=sender@company.com; dmarc=fail (p=REJECT) header.from=company.com
SPF Fail → DMARC Fail
The sending IP (198.51.100.23) is not in company.com's SPF record. DMARC fails because SPF alignment fails and DKIM isn't present. With p=REJECT this message will be blocked. Fix: add the IP or service to the SPF record.
MESSAGE BLOCKED
Received
from mail.thirdparty.com (thirdparty.com [198.51.100.23]) by mx.google.com; Mon, 20 May 2024 09:15:31 -0700
The relay's IP
198.51.100.23 is the IP that sent the message. This is the IP SPF checked — and it's not in company.com's SPF record. This is a relay that wasn't added to SPF. Common when adding new sending tools.
THIS IP FAILED SPF
Return-Path
<sender@company.com>
Return-Path
The envelope MAIL FROM domain is company.com. SPF checked this domain's record against the sending IP and failed. The SPF record for company.com doesn't list thirdparty.com's IPs.
SPF CHECKED THIS DOMAIN
From
Marketing Team <sender@company.com>
From
The visible From domain matches the Return-Path domain (company.com). Alignment would have passed IF SPF had passed. The fix is updating SPF, not the From address.
ALIGNMENT WOULD PASS IF SPF PASSED
FIX: Add the relay's IP or include:thirdparty.com to company.com's SPF TXT record.
Check current SPF: dig TXT company.com — count the lookups, must be under 10.
Email Headers — DKIM Body Hash Fail
Authentication-Results
dkim=fail (body hash did not verify) header.i=@company.com header.s=selector1; spf=pass smtp.mailfrom=company.com; dmarc=pass (p=QUARANTINE) header.from=company.com
DKIM Body Hash Fail
"Body hash did not verify" means the message body was modified AFTER DKIM signing. Common causes: URL rewriting by security gateways, disclaimer insertion, email signature tools modifying body. DMARC still passes here because SPF passed with alignment.
BODY MODIFIED IN TRANSIT
DKIM-Signature
v=1; a=rsa-sha256; d=company.com; s=selector1; bh=ORIGINAL_HASH_NO_LONGER_MATCHES; b=signature==
DKIM Signature — bh= mismatch
The bh= (body hash) was computed when the message was signed. Something modified the body between signing and delivery, so the hash no longer matches. The signature itself may be valid — the body changed.
BODY CHANGED AFTER SIGNING
Received
from proofpoint.company.com (proofpoint [10.0.0.50]) ... (added URL rewriting footer)
Security Gateway Hop
Security gateways like Proofpoint and Mimecast often rewrite URLs in the message body for click tracking and threat protection. This body modification breaks DKIM. If this is the cause, DKIM failure here is expected and DMARC passing via SPF is the fallback.
LIKELY CULPRIT
NOTE: This is the same reason Opensense strips DKIM. Modifying the body invalidates the existing signature. After Opensense processes the message and hands it back, the customer's mail system re-signs DKIM outbound. This is expected behavior — not a bug.
Email Headers — DMARC Alignment Fail
Authentication-Results
spf=pass (designates 198.51.100.55 as permitted sender) smtp.mailfrom=bounce@thirdparty.com; dmarc=fail (p=REJECT) header.from=company.com
DMARC Alignment Fail
SPF passes but DMARC fails. Why? The envelope MAIL FROM domain is thirdparty.com but the visible From is company.com. These don't align. SPF passes for thirdparty.com but that doesn't help company.com's DMARC. Fix: configure the third party to sign DKIM with company.com's domain.
ALIGNMENT MISMATCH
Return-Path
<bounce@thirdparty.com>
Return-Path — Wrong Domain
The envelope MAIL FROM domain is thirdparty.com. SPF passed for thirdparty.com — that domain authorized this IP. But thirdparty.com doesn't align with company.com in the From header. SPF alignment fails for DMARC purposes.
DOMAIN MISMATCH
From
Company Name <noreply@company.com>
From — The DMARC Reference Domain
DMARC checks whether the From domain (company.com) aligns with the SPF MAIL FROM domain (thirdparty.com) or the DKIM d= domain. Neither aligns here — no DKIM present, SPF is on a different domain. DMARC fails.
DMARC ALIGNS TO THIS
FIX: Configure DKIM signing on the third-party platform using company.com's domain. Add the DKIM CNAME records to company.com's DNS. Once DKIM d=company.com passes, DMARC alignment passes even if the envelope MAIL FROM is on a different domain.
04 — Opensense Mail Flow
What Opensense does and does NOT do
1
User sends email from Outlook or Gmail
Client hands message to M365 or Google Workspace via SMTP. DKIM is signed at this point by the customer's mail platform.
2
Connector routes message to Opensense
A mail flow rule or connector in M365/GWS routes outbound messages to Opensense over a secure TLS connection before they leave the tenant.
3
Opensense strips inbound DKIM signature
Opensense will modify the message body (add signature HTML, banner, etc.). A modified body invalidates the existing DKIM hash. Stripping it is correct behavior — a broken DKIM signature is worse than none.
4
Opensense processes and routes the message
Signature and banner are added. Message is handed back to the originating mail system (or to a configured relay like Zendesk, Front, or Outreach).
5
Customer's mail system re-signs DKIM and sends outbound
M365 or GWS applies a fresh DKIM signature to the modified message. SPF, DKIM, and DMARC are all evaluated at this outbound stage — not while Opensense was processing.
6
Recipient's server runs authentication checks
Authentication-Results are added. SPF validates the final sending IP. DKIM validates the re-signed signature. DMARC checks alignment. Opensense is not involved at this stage.
WHEN CUSTOMERS BLAME OPENSENSE: Opensense appears as a hop in Received headers. Admins who don't understand third-party mail flow see an unfamiliar server and assume it's the problem. In reality — Opensense does not control SPF, does not sign DKIM, and does not set DMARC policy. The customer's domain and mail platform control all of that.
05 — Troubleshooting Decision Tree
Ask these questions in order — every time
1Did the message send at all?
Yes — there's a sent record
Move to Step 2. Check headers.
No — nothing sent
Check SMTP settings, auth, network. Is the connector working?
2Is there an NDR (bounce)?
Yes — read the NDR
Note the SMTP code (550/421/etc.) and exact error text. That tells you who rejected it and why.
No bounce — went to spam
Check Authentication-Results. Check domain/IP reputation via MXToolbox Blacklist.
3Read the Received headers — did it travel the expected path?
Read bottom to top
Each hop = one server. Does Opensense appear? Does the path match the connector config?
Unexpected hops
A relay or security gateway in the path may be modifying the message or using an IP not in SPF.
4Check Authentication-Results — what passed and what failed?
All pass
Deliverability issue is reputation or content. Check blacklists, spam score, sending volume.
SPF fail
Check the sending IP against the SPF record. Is a relay not in SPF? Check lookup count (max 10).
5If DKIM failed — body hash or signature?
Body hash fail
Body was modified after signing. Check for security gateways, disclaimers, signature tools in the path.
Signature fail
Key mismatch. Look up selector._domainkey.domain in DNS. Does the public key exist and match?
6If DMARC failed — did alignment fail even though SPF or DKIM passed?
SPF passed but wrong domain
Envelope MAIL FROM domain doesn't align with From header domain. Third-party sender needs DKIM with the From domain.
Both SPF and DKIM failed
Fix the underlying SPF or DKIM issue first. DMARC needs at least one to pass with alignment.
